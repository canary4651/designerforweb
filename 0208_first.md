## 브라우저

    - 기본적으로는 하나만 가지고 작업을 함. 코드를 짤 때마다 브라우저에서 리로드 테스트 할 수 있음. 근데 보통 개발을 할 때 점유율이 노은 크롬으로 하고, 거의 끝날 때쯤 사파리를 열어서 대응을 시작함. 왜 깨지는 지 모르면 대응을 못하지만 알면 할 수 있음. 고정 폰트, 글꼴, 자간, 행간 등등.
    - 처음에 나는 여기까지를 지원해줄 거야. 라고 정하는 게 좋음. 그리고 크롬에서 개발하는 게 모든 도구가 더 편함.

- 물리적인(하드웨어) 크기와 소프트웨어 크기를 구분해야함.
- 수치를 고정으로 생각하면 안 됌. 여기에 이 요소가 위치한다는 레이아웃을 잡아야하지, 저 크기에 딱딱 있어야 한다는 건 불가능. (크롬이 달라질 수 있으니까)
- 세로 스크롤을 전제로 생각함. 가로는 스크롤바 때문에 생각할 때도 있음 (8px)

1. px - 고정 단위 / 아이콘 표현/ 44px/ 제한 값을 주고 싶을 때 많이 사용, 전통적인 방식이고 모든 곳에서 잘 동작한다는 기대가 있음. 테투리나 모서리 선. 
2. % : 상대 단위의 시작. 어디서 쓰느냐에 따라 그 값이 달라져서 어려움. 예시 : 2560x1440 에서 100% 라고 하면 가로 2560 세로 1440. 예를 들어서 700x400 에 있으면 100%가 700,400 인 거. 1%는 7. 
    - 마진, 패딩, 보더, 폰트 사이즈 요런데 넣기 시작하면 어려워짐.
    - %의 단점 : 헷갈리는 포인트가 생김. 박스를 하나 만들고 또 만들거나, 안에 박스를 넣을때..?!?! 상대 단위 계산이 어려움. 예를 들면 3개 박스를 동시에 띄위고 싶다 → 33.33333....%.... 그 안에 또 박스를 만든다면 다시 100%로 시작해야함 (박스 기준)

3. vh/vw : veiwport의 약자, 절대값이여서 짱 좋음. 어디서 1vh를 하든 14pt (1440 기준) - 뷰포트 기준이여서 박스 안에 박스를 만들어도 똑같음/ 단점은 ios에서 지원하기 힘듬 / 혁신적이다. 
- 위의 3을 상황에 맞게 많이 씀

4. em/ rem : 계산하기 어려워서 잘 안 씀.. 폰트 사이즈를 기준으로 함. 폰트 사이즈가 17px 이면, 1em은 17px이다. 폰트 사이즈를 기준으로 함. 폰트 사이즈를 유동적으로 주고 싶을 때 씀. 설계를 잘해야 함. 그래서 개발자들도 em이 계산하기 어렵다는 걸 깨달아서 rem을 만듬 (root 의 font size를 기준으로 함 - 1 rem 이 어디서는 17px가 고정: root는 html의 최상위 요소 )
